# Backtap Analyzer 仕様書 (MVP)

最終更新: 2026-02-22

## 1. 目的

スマホ画面のスクリーンショットをAIで解析し、まず以下を抽出する。

- 何のアプリが表示されているか
- 画面上の時間情報
  - 現在時刻
  - カウントダウン
  - いつから使えるか/解禁時刻 など

将来的に「記憶する系」の機能を追加する前提で、MVPは上記2項目を優先する。
特に Ingress のグリフハック用途を優先ユースケースとして扱う。

## 2. 想定ユーザー体験

- ユーザーがOS側トリガー（iPhone背面タップなど）で処理を起動
- スクリーンショット画像がアプリに渡される
- アプリが画像内容を解析し、構造化結果を返す
- 設定に応じて画像保存の有無を切り替える

## 3. 必須要件

- 解析対象
  - アプリ名推定 (`app_name`)
  - 時間情報 (`time_text`, `time_type`)
- 解析先切替（設定）
  - 端末内LLM / 端末内推論
  - ローカルLLM（自宅PCなどLAN上）
  - クラウドAPI
- 画像保存フラグ（設定）
  - 保存する/しない を切替可能

## 4. 技術制約 (重要)

### iOS

- アプリ単体で「他アプリ表示中の背面2タップ検知 + 直接スクショ取得」は不可。
- 実現方式は `背面タップ -> ショートカット -> アプリ連携` を前提とする。

### Android

- 背面タップの共通公開APIは前提にしない。
- 起動導線はクイック設定タイル/共有メニュー等で代替。
- 画面取得系はOS許可・通知要件が必要になる場合がある。

## 5. システム構成 (MVP)

- Flutterアプリ
  - 設定UI
  - 解析結果表示
  - 解析履歴の土台
- OCR
  - iOS: Vision OCR (ネイティブ)
  - Android: 後続で同等導線を実装
- 解析エンジン
  - On-device: OCR文字列のルール解析（MVP）
  - Local/Cloud: HTTP API委譲

## 6. データ仕様

### 入力

- スクリーンショット画像
- OCR抽出テキスト

### 出力 (標準JSON)

```json
{
  "app_name": "YouTube",
  "time_text": "12:34",
  "time_type": "clock",
  "confidence": 0.93
}
```

### `time_type` 列挙

- `clock`: 時刻表示
- `countdown`: 残り時間表示
- `available_at`: 解禁時刻/開始時刻
- `unknown`: 判定不能

## 7. 設定項目

- `ai_target`: `on_device | local_api | cloud_api`
- `local_api_url`
- `cloud_api_url`
- `cloud_api_key`
- `save_screenshot`: `true | false`

## 8. MVP範囲 (今回)

- 画像選択 -> OCR -> 解析 -> 結果表示
- 解析先切替と設定保存
- 時間情報の基本抽出ルール

## 9. 次フェーズ

- iOSショートカット/App Intent 連携本実装
- Android起動導線の統一
- 記憶機能（履歴保存・検索・再利用）
- JSONスキーマ検証とエラー耐性強化

## 10. Ingressグリフハック拡張 (記憶機能 v1)

### 10.1 目的

- 覚えた図形（グリフ）を入力または画像解析から認識する
- 認識結果を記憶し、通知で素早く再確認できるようにする

### 10.2 入力方式

- スクリーンショット解析でグリフ候補を抽出
- 手書き入力（後続実装）でグリフ候補を抽出

### 10.3 記憶データ

- `glyph_id`: 内部識別子
- `label`: 表示ラベル（例: `GLYPH_A`）
- `meaning`: 意味
- `confidence`: 認識確信度
- `learned_at`: 学習日時
- `source`: `screenshot | handwriting`
- `image_path`: 参照画像（任意）

### 10.4 通知要件

- 通知から「ぽこっと見て確認」できることを重視する
- OS通知で外字フォント表示は崩れる可能性があるため、MVPは以下を標準にする
  - 通知本文: `label` と `meaning`
  - 通知添付: グリフ画像（可能な端末のみ）
- 通知タップで詳細画面に遷移し、正誤入力で記憶を更新できること

### 10.5 成功条件

- 登録済みグリフに対して通知から意味を即時確認できる
- 正誤入力で `confidence` が更新される
- 同一グリフの重複登録を抑制できる

## 11. AI利用制限解析拡張 (運用管理 v1)

### 11.1 目的

- 各AIサービス画面の「何時まで使用不可」「次回利用可能時刻」を読み取り、分かりやすく管理する
- UTC/JSTなど異なるタイムゾーン表記をユーザーのローカル時刻に正規化して表示する

### 11.2 解析対象

- 制限メッセージ（例: `Try again at ...`, `Available in ...`, `Rate limit exceeded`）
- 時刻表記（`UTC`, `JST`, `PT`, `ET`, 相対時間表記）
- 画面固有の特徴（ロゴ、見出し、ボタン文言、UI配置）

### 11.3 出力データ (追加)

- `restriction_type`: `rate_limit | daily_cap | temporary_block | unknown`
- `available_at_utc`: ISO8601 UTC時刻（判定できた場合）
- `available_at_local`: ユーザーローカル時刻（表示用）
- `remaining_duration_sec`: 相対時間が取れた場合の秒数
- `source_timezone`: 画面上で検出したタイムゾーン
- `normalized_timezone`: 変換後タイムゾーン（通常は端末ローカル）
- `screen_fingerprint`: 画面特徴ハッシュ
- `service_name`: 推定サービス名（例: ChatGPT, Claude, Gemini）

### 11.4 画面特徴学習

- 画像とOCRテキストから `screen_fingerprint` を生成し、同系統画面を同一クラスタとして扱う
- ユーザーが手動修正したラベル（サービス名・制限種別）を学習データとして保存
- 次回同種画面では過去の修正を優先適用し、推定精度を上げる

### 11.5 UI/通知要件

- 一覧で「今受けている制限」をサービス別に表示する
- 解除見込み時刻はローカル時刻で表示し、元のタイムゾーンも併記する
- 解除時刻が近い順にソート可能
- 必要に応じてローカル通知で再開可能タイミングを知らせる

### 11.6 成功条件

- UTC/JST等の表記差があってもローカル時刻で統一表示できる
- 同じサービスの同種制限画面を再解析したとき、前回より高精度に分類できる
- 「現在受けている制限」が一覧で把握できる

## 12. Community Game Pack共有拡張

### 12.1 目的

- ユーザーが自分で作成・調整した解析設定を `Game Pack` として出力できる
- LINE等で共有された `Game Pack` を他ユーザーが取り込んで再利用できる

### 12.2 Packの基本構造

- `pack_id`
- `pack_name`
- `service_name`
- `version`
- `rules`（キーワード/正規表現/時刻解釈ルール）
- `timezone_rules`
- `display_template`
- `author_name`（任意）

### 12.2.1 表示モデル標準

- Pack内の表示定義は固定 `WidgetType` を使う
- 初期サポート:
  - `list`
  - `bar`
  - `countdown`
  - `checklist`
  - `table`
- 各Widgetは共通フィールドを持つ
  - `title`
  - `value`
  - `unit`
  - `status`
  - `deadline`（任意）
  - `progress`（0.0〜1.0、任意）
- 方針: 「データは共通JSON、見た目はプリセット選択」でユーザーがレイアウト設計しなくても使える形にする

### 12.2.2 フォーマット方針

- 内部標準: アプリ専用の固定JSON schema
- 共有互換: テキスト配布しやすいJSONをそのまま共有
- 将来拡張: Adaptive Cards互換エクスポートを任意機能として追加可能にする

### 12.3 セキュリティ要件

- Packはデータのみ（JSON）とし、任意コード実行は不可
- インポート時にスキーマ検証を必須化
- `api_key` など秘匿情報はエクスポート対象外
- 外部URL呼び出しやファイル操作など危険操作を禁止
- 未検証Packには警告表示を行う

### 12.4 エクスポート要件

- 「自分の設定をPackとして出力」をサポート
- 出力時に個人情報・機密情報が含まれないことを確認
- バージョンと更新日時を自動付与

### 12.5 インポート要件

- 取り込み前に差分プレビュー（追加/更新されるルール）を表示
- 既存設定との競合時は「上書き/併存」を選択できる
- 失敗時は原因（形式不正、禁止項目）を明確に表示

## 13. ダッシュボード/進捗管理拡張

### 13.1 目的

- アプリ起動直後に、管理中の対象を一覧で把握できる
- 「解除まであと何分」「次に何を何回やるべきか」を即判断できる

### 13.2 ホーム画面要件

- 管理リストをカード形式で表示
- 各カードに以下を表示
  - `service_name` / `game_name`
  - `status`（制限中/解除済み/不明）
  - `unlock_in_min`（解除まであと何分）
  - `next_available_at_local`（ローカル時刻）
  - `progress_percent`（達成率）

### 13.3 目標ペース表示

- 日次目標を設定可能
  - `daily_target_count`（1日に必要な回数）
- 現在進捗から、残り時間内に必要な実行ペースを計算して表示
  - `required_runs_today`（今日残りで必要な回数）
  - `required_runs_per_hour`（1時間あたり必要回数）
- 文言例
  - 「明日までに達成するには、残り本日 `N` 回必要」
  - 「終了まで `X` 時間、1時間あたり `Y` 回ペース」

### 13.4 ソート/フィルタ

- `unlock_in_min` が短い順で並べ替え
- `status=制限中` のみ表示
- `game/service` 別フィルタ

### 13.5 成功条件

- アプリを開いた時点で、優先すべき対象が1画面で分かる
- 解除見込み時間と残作業量を同時に確認できる
- 日次目標に対して「あと何回必要か」を誤解なく表示できる

## 14. 非機能要件

- APIキーは安全に保存（Keychain/Keystore想定）
- 解析失敗時の再試行導線
- 低遅延（MVP目標: 1〜3秒程度、ネットワーク依存）

## 15. 受け入れ条件 (MVP)

- スクリーンショット1枚を入力して、以下が返ること
  - `app_name` が文字列で返る
  - `time_text` が文字列で返る
  - `time_type` が列挙値で返る
- 設定で解析先を切替できること
- 画像保存ON/OFFを切替できること
